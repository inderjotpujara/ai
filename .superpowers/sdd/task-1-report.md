## Issues found and fixed (beyond the brief's literal steps)

These surfaced only when running the **root** gates (typecheck / lint /
full `bun run test`), not from the `web/`-local gates, so they were easy to
miss — flagging clearly since the brief didn't call them out.

### 1. `@ai-sdk/provider-utils` silently lost its root hoist (root typecheck broke)

`src/core/agent-def.ts` and `src/core/agent.ts` do `import type {
ProviderOptions } from '@ai-sdk/provider-utils'` directly, but root
`package.json` never declared it as an explicit dependency — it was only
present in `node_modules/@ai-sdk/provider-utils` because Bun's hoisting
algorithm happened to promote it there as a side effect of `@ai-sdk/react`
(root devDependency) sharing a compatible version range with `ai`'s own
transitive dependency of the same name. The instant `@ai-sdk/react` moved
to `web/package.json` (this task's whole point), Bun's hoisting decision
changed and `@ai-sdk/provider-utils` was no longer hoisted to root
`node_modules` at all — only nested under `.bun/@ai-sdk+provider-utils@*`
per-dependent. Result: `bun run typecheck` at root failed with `Cannot find
module '@ai-sdk/provider-utils'` in both files.

**Fix:** added `"@ai-sdk/provider-utils": "^4.0.34"` as an explicit root
`dependencies` entry (matching the version `ai@6.0.217` itself depends on).
This is a real, pre-existing latent bug in the dependency manifest (relying
on incidental hoisting for a directly-imported package) that this task's
dependency relocation exposed. After the fix, `bun install` hoists
`node_modules/@ai-sdk/provider-utils@4.0.38` (the version selected by
`@ai-sdk/react`'s own tighter range, still satisfying `^4.0.34`) and root
`bun run typecheck` is clean again.

### 2. Root `bun test` (native runner) auto-discovered `web/src/test/harness.test.tsx` (root test gate broke)

The whole premise of this task — per the launch brief — is that `web/`'s
React/DOM/Vitest tests "cannot ride the repo's Bun-native `bun test` gate."
But Bun's native test runner recursively discovers any `*.test.{ts,tsx}`
file in the whole working tree by default; it does **not** respect
`tsconfig.json`'s `exclude` or Biome's `files.includes`. So plain `bun test`
from root found `web/src/test/harness.test.tsx` and tried to run it under
Bun's own (non-happy-dom, non-Vitest) test environment — `document` is
undefined there, so both harness tests failed:
```
ReferenceError: document is not defined
  at render (.../@testing-library/react/dist/pure.js:256:19)
(fail) web test harness > renders a DOM node and jest-dom matchers work
(fail) web test harness > runs under a cross-origin-isolation-aware DOM
```
This would have broken the extended `check` script's final `bun run test`
step for every future `web/` test file, silently defeating the entire
purpose of this task (decoupling `web/`'s toolchain from bun-native test).

**Fix:** changed root `test`/`test:file` scripts to `bun test
--path-ignore-patterns 'web/**'` (verified CLI flag on bun 1.3.11). Root
`bun run test` is back to the pre-task baseline: 1161 pass / 36 skip / 0
fail across 294 files (was 295 files / 2 extra fails with `web/`
undiscounted). `web/`'s own tests still run correctly and exclusively under
Vitest via `check:web` / `cd web && bun run test`.
