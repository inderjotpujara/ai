## 1. Thesis

Phases 1–7 shipped every functional surface of the local web UI — streaming chat, run history + live trace, Crews/Workflows browse-run-watch, Builders + Library, session persistence + recall, and browser-native voice dictation — but every phase deferred its own polish, and Phase 8 is where those IOUs come due. This phase closes every remaining a11y, correctness, and observability gap so the 30b capability itself can flip 🟡→✅, with a **single documented exception carried forward as future scope, not debt**: barge-in / interruptible voice. The parent spec's own build-order item 8 (quoted above) still lists "voice barge-in" as a live-verify criterion — inherited from before Phase 7 locked voice to **dictation-only** (Phase 7 D2, `2026-07-18-slice-30b-phase7-voice-design.md` §2). This phase corrects that inherited criterion across spec/README/ROADMAP rather than quietly building the thing Phase 7 deliberately rejected. **A11y target: WCAG 2.1 AA.**

Two data points anchor how much ground this phase covers: the app currently has **zero `<label>` elements and zero `aria-pressed` attributes anywhere in `web/src`** (verified by grep), and there is **no client→server telemetry channel of any kind** (no `/api/telemetry` route, no `navigator.sendBeacon` call; the only client-side sink is a `console.error` in `web/src/shared/ui/error-boundary.tsx:15`, explicitly commented "a telemetry sink lands in a later phase" at line 14).

## 2. Scope (locked with the user 2026-07-19)

| Decision | Choice |
|---|---|
| A11y target | WCAG 2.1 AA |
| Interim-transcript semantics | **Progressive reveal after capture** — words appear as Moonshine decodes the already-captured buffer (post hold-release / VAD end-of-speech). **Not** real-time-during-speech (explicitly rejected, §9). |
| kind=agent notification fix | Chat turns get their own root span (`chat.run`), not a NOTIFIABLE_KINDS filter change — see D9. |
| Telemetry channel | New schema-validated `POST /api/telemetry` + client `navigator.sendBeacon`, feeding one new `voice.transcribe.web` span. |
| ⌘K model | `Command.run` widened to also support non-navigation actions, not nav-only. |

**Scope ships as 5 reviewable increments** (repo SDD increment pattern):

1. **A11y foundations (app-wide, non-voice)** — focus-visible ring, real labels, `aria-pressed`/`aria-label`, tab keyboard pattern, a JS-motion reduced-motion hook, `vitest-axe` harness.
2. **Voice a11y + interim ASR + audio fidelity** — `aria-live` status announcements, progressive-decode-reveal transcript, downsampler anti-alias filter.
3. **⌘K command-palette completeness** — action commands, voice/theme toggles, `go-agents`, dedupe the degenerate jump-to-* set.
4. **Correctness + observability** — the chat-turn `kind=agent` mis-notification fix; a new client telemetry beacon.
5. **Docs + live-verify + land** — all four surfaces + ledger + Artifact regen, criterion reconciliation, gated live-verify, full-slice landing.

**Explicitly out of scope (future, NOT deferred debt):** barge-in / interruptible voice + TTS voice-out (rejected by Phase 7 D2); real-time during-speech ASR (progressive-decode-reveal is this phase's answer instead, D6).

## 3. Design decisions (D-series)

- **D1 — A11y foundations: one design-token-driven focus ring, real labels, `aria-pressed`, landmark labels.** Remove the composer's `focus:outline-none` (`web/src/shared/ai-elements/prompt-input.tsx:53`) in favor of an app-wide `:focus-visible` ring sourced from a new design token (alongside the existing light/dark token set). Add real `<label>`/`htmlFor` pairs for the composer `<textarea>` (`prompt-input.tsx:46-54`) and the Settings model-tier `<select>` (`web/src/features/settings/index.tsx:162-173`) — currently the ONLY form controls in the app with zero `<label>` coverage — plus a new `sr-only` utility class (none exists today) for visually-hidden label text. Add `aria-pressed` to every toggle button that lacks it: the theme toggle (`web/src/app/app-shell.tsx:82`), the voice-input and OS-notify toggles (`settings/index.tsx:136-160`). Add `aria-label` to the three unnamed `<aside>` landmarks: `features/sessions/index.tsx:38`, `features/workflows/workflow-detail.tsx:111`, `features/runs/waterfall.tsx:53`.
- **D2 — Library/Builders tabs get a real keyboard pattern, not just ARIA roles.** Both `LibraryArea` (`web/src/features/library/index.tsx:28-49`) and `BuildersArea` (`web/src/features/builders/index.tsx:22-33`) already set `role="tablist"`/`role="tab"`/`aria-selected` but have no `role="tabpanel"`, no `aria-controls` linkage, and no arrow-key roving-tabindex navigation — a screen-reader user can identify the tabs but a keyboard-only user cannot move between them without Tab-cycling every other control first. Add the standard tab widget pattern (arrow-key roving `tabIndex`, `aria-controls`/`id` linking each tab to its panel, `role="tabpanel"`) to both, sharing one small helper rather than duplicating the wiring twice.
- **D3 — JS/library motion needs its own reduced-motion gate; the existing CSS rule doesn't reach it.** `web/src/shared/design/tokens.css:59-66`'s `@media (prefers-reduced-motion: reduce)` block only zeroes CSS `animation-duration`/`transition-duration` — it has no effect on `@xyflow/react`'s imperative `fitView`/pan/zoom calls in `DagView` (`web/src/shared/dag/dag-view.tsx:138`), which run via JS regardless of the CSS rule. New `web/src/shared/a11y/use-reduced-motion.ts` reads `window.matchMedia('(prefers-reduced-motion: reduce)')` and lets `DagView` skip/shorten `fitView`'s animated transition when true.
- **D4 — `vitest-axe` is the automated a11y regression net.** `@testing-library/jest-dom` is already a dependency (`web/package.json:44`); no axe integration exists today. Add `vitest-axe`, wire a `toHaveNoViolations` matcher, and add baseline no-violations assertions across the app's key screens (Chat, Sessions, Runs, Library, Builders, Settings) — catching regressions in D1–D3's fixes going forward, not a one-time audit.
- **D5 — Voice status changes need `aria-live`, not just visual text.** `MicButton` (`web/src/features/voice/mic-button.tsx`) today announces nothing but the error state (`role="alert"` at line 110); the "Loading voice model…" span (lines 104-108), the "● Listening" label (line 83), and the interim-transcript span (lines 96-103) are all silent to assistive tech. Wrap the status region in `aria-live="polite"` so `loading`/`listening`/`transcribing` transitions (the `VoiceStatus` union, `use-voice-input.ts:10-16`) are announced without adding a second visual affordance.
- **D6 — Progressive decode reveal via a `TextStreamer` callback, explicitly NOT real-time-during-speech.** `stt.worker.ts`'s `transcribe()` (lines 162-175) currently calls `asrModel.generate({...inputs, max_new_tokens: 256})` and returns the batch-decoded text only once generation fully completes. Add a transformers.js `TextStreamer` callback to that `generate()` call, emitting a new `{ kind: 'transcribeInterim'; id: number; text: string }` variant added to the `SttWorkerResponse` union (`stt.worker.ts:35-40`, alongside the existing `transcribeResult`). `stt-engine.ts` (which today only handles `transcribeResult`, `stt-engine.ts:73`) forwards interim chunks through a new `onInterim` callback, and `use-voice-input.ts` replaces its `setInterim('…')` placeholder (line 168) with the real streamed text — the final transcript still arrives via `onFinal` (line 181) unchanged. **Semantics, stated plainly:** interim words appear as Moonshine *decodes the already-captured buffer* after the segment closes (hold-release or VAD end-of-speech) — never while the user is still speaking. Real-time-during-speech streaming ASR was considered and rejected: Moonshine's encoder needs the full utterance before decoding starts, so it would require an entirely different streaming-capable model, out of scope for a polish phase.
- **D7 — Downsampler gains an anti-alias low-pass filter; stays a zero-dependency module.** `web/src/features/voice/downsampler.ts` (module doc, lines 3-28) implements bare two-point linear interpolation (the actual conversion at lines 50-57) with no anti-aliasing stage — a textbook aliasing risk when downsampling 48kHz→16kHz. Add a simple one-pole low-pass filter (cutoff ~7.5kHz, half the target Nyquist) ahead of the existing interpolation, with its own carried filter state reset in `flush()` (mirroring the existing `nextP`/`globalOffsetSoFar`/`prevLast` reset pattern at lines 70-73). The module stays dependency-free so both `audio-capture.ts` and a future worklet build inherit it identically.
- **D8 — ⌘K: widen `Command` to support actions, not just navigation; fill the two named gaps.** `Command` (`web/src/app/commands.ts:5-9`) is `{ id, label, run: (nav) => void }` — nav-only by construction, and the file's own comment (lines 11-14) explicitly defers "switch-model" and action commands to "Phase 8". Add a second command kind whose `run` takes no `nav` argument (an action callback), and a small palette-side dispatch that calls whichever shape a command has. Add: a voice-input toggle (setting already exists, `VOICE_ENABLED_KEY` in `settings/index.tsx:6`), a theme/dark-mode toggle (state already lives in `app-shell.tsx`), and a `go-agents` nav command (every other stub area — Crews, Workflows, Builders, Runs, Library, Settings — already has one; Agents does not). The degenerate duplicate `jump-to-run`/`jump-to-crew`/`jump-to-workflow`/`jump-to-sessions`/`search-sessions` entries (`commands.ts:39-64`, today all five just re-navigate to their list route with no differentiation) get deduped into fewer, real entries, plus a genuinely new "jump to a recent run" command that deep-links to a specific `runId` rather than the bare list.
- **D9 — kind=agent notification fix: give chat its own root span; do NOT touch `NOTIFIABLE_KINDS`.** Today `withRunSpan` (`src/telemetry/spans.ts:225-239`) unconditionally opens an `inSpan('agent.run', ...)` root (line 230), and it has exactly **one caller in the whole codebase**: `src/cli/run-chat.ts:21`, which is the shared chat-turn path both the CLI and `POST /api/chat` run through. So every `agent.run`-rooted run in the repo today is, in fact, a chat turn — `deriveRunKind` (`src/run/run-dto.ts:54-65`) reads that root at line 57 and returns `RunKind.Agent`, and the web notifier's `NOTIFIABLE_KINDS = {Crew, Workflow, Agent}` (`web/src/features/notifications/notify-diff.ts:18-22`) fires a completion toast for a long *chat* turn — violating the invariant documented at `notify-diff.ts:16-17` ("a chat turn never fires one"). **The fix is architectural, not a filter tweak:** `withRunSpan` is a generically-named helper (its doc comment makes no chat-specific claim) that some *future* standalone-agent-run feature could reuse to legitimately open an `agent.run` root — narrowing `NOTIFIABLE_KINDS` to exclude `Agent` would silently mute that future feature too. Instead, `run-chat.ts` gets its own root-span opener (a `chat.run`-naming sibling of `withRunSpan`, same attribute-setting body) so chat turns stop claiming the `agent.run` name at all; `deriveRunKind` gains an explicit `chat.run → RunKind.Chat` branch (functionally already covered by its existing default-to-Chat fallback at line 64, added anyway for the doc comment's own "recognizes X/Y/Z" listing to stay accurate); and `RUN_ROOT_NAMES` (`run-dto.ts:36-45`, which `runRootSummary` uses to find a run's lifecycle-deciding root) gains `'chat.run'` alongside the existing `'agent.run'` entry — omitting this second edit would make every chat run read as perpetually "Running" in `GET /api/runs` (the exact ghost-row failure mode `RUN_ROOT_NAMES`'s own doc comment, lines 27-35, warns about). `NOTIFIABLE_KINDS` itself is unchanged. Historical on-disk chat runs (pre-Phase-8) stay `agent.run`→`RunKind.Agent` — a one-time, documented, accepted classification quirk for old data, not a live bug.
- **D10 — Voice telemetry beacon: a new, distinctly-named span, not a repurposed existing one.** A `voice.transcribe` span already exists (`src/telemetry/spans.ts:1053-1055` region, attributes `ATTR.VOICE_STT_MODEL`/`VOICE_CAPTURE_SOURCE`/`VOICE_AUDIO_SECONDS`/`VOICE_DURATION_MS`/`VOICE_OUTCOME`) but it is emitted **server/CLI-side**, in-process, for the Slice-29 CLI transcribe path — it has no browser caller and different attributes than a browser beacon would carry (no `wordCount`/`realTimeFactor`/`engine`). Rather than overload that span's shape with a second, differently-attributed call site, this phase adds a new `voice.transcribe.web` span, written server-side by a new schema-validated `POST /api/telemetry` route (new `src/server/telemetry/`, wired in `src/server/app.ts` alongside the existing `json()`/`ISOLATION_HEADERS` conventions) that the browser reaches via `navigator.sendBeacon` — the one delivery mechanism that survives a page unload/navigation, which a completed voice transcription can trigger if the user immediately sends the message. A new contract, `src/contracts/telemetry.ts` (`TelemetryEventSchema`), validates the beacon body. Client emitter: `web/src/shared/telemetry/beacon.ts`, invoked from the transcribe-completion path in `use-voice-input.ts` (the same `.then()` block at lines 169-183 that already calls `opts.onFinal`).

- `src/telemetry/spans.ts` — new `withChatRunSpan`-style helper opening `inSpan('chat.run', ...)` (D9), plus the new `voice.transcribe.web` span writer (D10).
- `src/cli/run-chat.ts:21` — calls the new chat-specific span opener instead of `withRunSpan`.
- `src/run/run-dto.ts` — `RUN_ROOT_NAMES` (lines 36-45) and `deriveRunKind` (lines 54-65) both gain `'chat.run'` (D9).

### 4.6 Data flow

**Progressive decode:** hold-release/VAD-end-of-speech closes a segment (unchanged from Phase 7) → `engine.transcribe(frames)` starts → the worker's `TextStreamer` callback fires `transcribeInterim` messages as Moonshine decodes token-by-token → `stt-engine.ts` relays each to `use-voice-input.ts`'s `onInterim` → the composer's interim display updates in place → the final `transcribeResult` replaces it via the existing `onFinal` path, unchanged.

**Telemetry beacon:** on transcribe completion (same call site as the existing `onFinal`), `use-voice-input.ts` computes `durationMs`/`wordCount`/`realTimeFactor` and calls `beacon.ts`'s `navigator.sendBeacon('/api/telemetry', ...)` — fire-and-forget, survives the page unloading before the request completes → server validates + writes `voice.transcribe.web`.

**kind=agent fix:** a chat turn's root span is now `chat.run`, not `agent.run` → `deriveRunKind` returns `RunKind.Chat` → `NOTIFIABLE_KINDS` (unchanged) doesn't match → the poll-diff hook (`notify-diff.ts`, Phase 6, untouched) never queues a toast for it. A crew/workflow/(future) standalone-agent run keeps its own distinct root and keeps notifying exactly as before.

## 7. The hard part

### 7.1 Progressive-decode-reveal correctness

The `TextStreamer` callback fires from inside the worker's async `generate()` call, and its interim messages must survive the request being superseded — Phase 7's `use-voice-input.ts` already tracks a `validSegmentersRef` set (not a single latest-wins token, `use-voice-input.ts:69-79`) precisely because a graceful stop must let an in-flight transcribe's *final* still land even after a newer gesture starts. Interim messages need the same discipline: each must be correlated to its originating `transcribe` request by numeric `id` (the existing request/response id-matching `stt-engine.ts` already does for `detectSpeech`/`transcribe`), ordered, and cleanly superseded by that same request's final result — never bleeding into the NEXT segment's interim display if a fast double-tap starts a new gesture while the previous segment's decode is still streaming. Both hold-to-talk and VAD-gated tap-to-toggle gestures must feed this correctly, since they hand frames to `engine.transcribe()` through the same code path (`use-voice-input.ts:161-196`).

**Requirements the review must adversarially verify:** (a) interim messages for a superseded/invalidated segmenter (per the existing `validSegmentersRef` gate) are dropped, never displayed; (b) interim text is monotonically replaced, never appended-then-replaced-with-a-shorter-string (a decode restart artifact); (c) the final `transcribeResult` always wins over any late-arriving interim for the same request id; (d) a back-to-back gesture (new segment starts before the previous segment's decode finishes) never shows the new segment's interim text as if it were the old segment's, or vice versa.

### 7.2 kind=agent fix must not regress either direction

The fix touches three files that together decide what a `GET /api/runs` row's `kind` is and whether it ever notifies: `spans.ts` (which root span name gets opened), `run-chat.ts` (which opener chat calls), and `run-dto.ts` (both `RUN_ROOT_NAMES`, which decides whether `runRootSummary` finds a real lifecycle/duration at all, and `deriveRunKind`, which decides `Chat` vs `Agent`). Missing the `RUN_ROOT_NAMES` edit specifically would not just misclassify chat's `kind` — it would make every chat run report `lifecycle: Running` forever (the exact ghost-row failure the set's own doc comment warns about), a strictly worse regression than the bug being fixed.

**Requirements the review must adversarially verify:** (a) a `chat.run`-rooted trace's `runRootSummary` resolves a real `lifecycle`/`durationMs`, not a perpetual-Running ghost; (b) `deriveRunKind` returns `RunKind.Chat` for a `chat.run` root and (synthetically, since no shipped feature exercises it yet) still returns `RunKind.Agent` for a bare `agent.run` root, proving `withRunSpan`'s generic `agent.run` capability is intact for whatever future caller needs it; (c) `NOTIFIABLE_KINDS` is byte-for-byte unchanged (`notify-diff.ts:18-22`); (d) a long chat turn in a real browser genuinely does not toast, and a crew/workflow run genuinely still does (live-verify, increment 5).

## 9. Forward-items (deferred, tracked)
