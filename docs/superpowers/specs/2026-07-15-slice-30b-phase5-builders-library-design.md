## 1. Thesis

Phases 1–4 gave the web UI a real chat surface, a Runs history/trace browser, and Crews/Workflows browse-run-watch. Every one of those phases exposed an engine capability that **already existed** — the web layer is a thin BFF over real code, never a reimplementation. Phase 5 is the same move applied to the **last four unexposed engines**: the agent-builder (`src/agent-builder/`), the crew/workflow-builder (`src/crew-builder/`), the provisioning downloader (`src/provisioning/`), the MCP mount registry (`src/mcp/`), and the memory/RAG store (`src/memory/`). **Nothing new is built at the engine layer this phase** — every capability below already has a CLI or a programmatic entry point; Phase 5 gives each one a web seam.

The genuinely new build is:
1. **A transport split by interactivity** — three different wire shapes for three different kinds of engine work (§3, D1–D3), rather than forcing everything through one pattern.
2. **Two consumers of previously-unconsumed Phase-2 scaffolding** — `ConsentRegistry` (built in Phase 2, never wired to a real engine call) becomes the confirm/reuse channel for the builder's mid-flow consent and MCP's mount consent; `StatusEventType.Confirm`'s `kind` field, whose own doc comment already lists `build`/`reuse`/`mcp-mount` as anticipated free-string values (`src/contracts/events.ts:54-64`), gets its first real callers.
3. **A pull→spans bridge** — the one place a non-interactive fire-and-watch flow (a model pull) needs *live* progress, not just a post-hoc trace, solved by making each `DownloadProgress` tick its own short-lived child span (§7.2) rather than inventing a second live-progress channel.
4. **A thin, per-tab web layer** — `/builders` (two guided wizards) and `/library` (one area, three tabs: Models · Memory · MCP), each a small REST/SSE consumer over the seams above.

This ships as **one Phase-5 milestone**, built in 6 reviewable increments (§5). Like every prior 30b phase, it is a **partial-slice landing**: README/ROADMAP mark Phase 5 landed; the 30b capability itself stays unflipped until Phases 6–8 close voice, persistence, and accessibility.

## 3. Design decisions (D-series)

- **D1 — Interactive flows ride chat-style SSE + the consent channel.** A build both streams progress and awaits mid-flow consent (reuse-offer, commit) from a human; an MCP test-mount awaits mount consent and may pop an OAuth tab. Both use the Phase-2 `createUIMessageStream` + `writer.merge` + `createUIMessageStreamResponse` shape (`src/server/chat/handler.ts`) and the `data-confirm` event (`StatusEventType.Confirm`, already schemed). These flows also mint a run (`newRunId` + `createRun`) and bind telemetry via `withRunContext` so their spans land in `runs/<id>/spans.jsonl` and the run is visible in the Runs browser **after** it finishes — the root span (`agent.build`/`crew.build`) only exports when it closes (§7.1), so this is retrospective visibility, not live-tailing.
- **D2 — Consent-resolved-upfront flows are fire-and-watch, unchanged from Phase 4.** A model pull has already collected every consent it needs (model selection, disk-shortfall) in the UI before the request fires, so `POST /api/models/pull` mints a `runId`, pre-creates the run dir (`createRun`), detaches the download (`void deps.runModelPull(...).catch(→error.json)`), and returns `{ runId }` at once — the exact shape `handleCrewRun` already established (`src/server/crews/run.ts:41-68`). The browser opens the **existing** `GET /api/runs/:id/stream`; no new stream handler.
- **D3 — Simple queries are plain REST.** Lists (installed+pullable models, MCP servers, memory spaces) and stats ride `apiFetch` (`web/src/shared/contract/client.ts`), zod-validated, no cursor (small in-memory/on-disk sets, mirroring D9 from Phase 4).
- **D4 — `ConsentRegistry` gets its first real caller.** `src/server/consent/registry.ts`'s `port: ConfirmPort` (mint a promptId, emit `data-confirm`, resolve via `POST /api/runs/:id/respond`) was built in Phase 2 and wired into `ServerDeps` at boot, but **no engine call has ever invoked it** — `respond.ts` only reads it. Phase 5 wires it as: `BuilderDeps.confirm` (kind `'build'`), `BuilderVerifyDeps.confirmReuse` (kind = the `ReuseKind` value, `'reuse'`/`'offer'`), and MCP's `ConsentDeps.ask` (kind `'mcp-mount'`) — all three kinds are already named in the `ConfirmEventSchema` doc comment (`src/contracts/events.ts:54-58`). `respond.ts`'s `_runId` param is documented as informational-only (the registry is server-wide, keyed by an unguessable `promptId`) — which is exactly what lets one registry safely serve concurrent builder/mcp/(future) confirms without per-run scoping work.
- **D5 — The builder/crew-builder proposal types are already JSON-safe.** Unlike `CrewDef`/`WorkflowDef` (Phase 4's D1, which carry `ToolSet`/`ZodType`/closures and need real projection), `AgentProposal` (`src/agent-builder/types.ts:11-18`) and `CrewIR`/`WorkflowIR` (`src/crew-builder/ir.ts`) are **already zod schemas with no closures, no `ToolSet`, no `ZodType`** — every field (`name`, `description`, `systemPrompt`, `modelReq`, `suggestedServers: {packName, scopeToAgent}[]`, `rationale` for agents; `members`/`tasks`/`process` for crews; `steps` for workflows) is plain data. `AgentProposalDTO`/`CrewProposalDTO`/`WorkflowProposalDTO` are therefore near-identity re-exports through a contracts-owned zod schema, not lossy projections — the simplest DTO work in the whole slice.
- **D6 — The proposal preview reuses `DagView` (Phase 4's D7), not a new component.** `CrewIR`/`WorkflowIR`'s `members`/`tasks`/`steps` shape is structurally close to Phase 4's `CrewDetailDTO`/`WorkflowDetailDTO`, so the same `crewGraph`/`workflowGraph`-style derivation (edges via `dependsOn`, else previous-in-order — the same `effectiveDeps` lineage) turns a staged (not-yet-committed) proposal into a `DagModel` for `<DagView>`. Nodes render with a new `DagStatus.Proposed` member (`web/src/shared/dag/types.ts`) so an unstaged node reads visually distinct from a live/committed one. The agent-builder's proposal (a single agent + its `suggestedServers`) gets a small 2-tier graph (agent node → suggested-server nodes) — the same `DagView` component, a much simpler `DagModel`.
- **D7 — The pull→spans bridge is per-tick child spans, not events on one long span.** `JsonlFileExporter` (`src/telemetry/jsonl-exporter.ts:58-71`) appends a span to `spans.jsonl` only when **that span itself ends** — confirmed by inspecting `withAgentBuildSpan`/`withProvisionSpan` (`src/telemetry/spans.ts`), which wrap their whole body in ONE `inSpan(...)` call and record everything else as `span.addEvent(...)` (accumulated in-memory, invisible until the wrapping span closes). A model pull needs the opposite: **live** progress in the existing poll-based `/api/runs/:id/stream`. The fix (detailed in §7.2) is to open+close a **new short-lived child span per `DownloadProgress` callback**, nested under a long-lived `model.pull` root that stays open for the whole download — mirroring how `withStepSpan` children (not the long-lived `crew.run`/`workflow.run` root) are what make Phase-4 live-watch actually incremental.
- **D8 — Every engine-touching route mints its own ephemeral run, even non-streaming ones.** So that memory ingest/recall and MCP add/test-mount are observable in the Runs browser (the standing telemetry note), a plain `POST /api/memory/:space/ingest` (etc.) also mints `newRunId()` + `createRun()` + binds `withRunContext` around the one call to `store.ingest(...)`/`recall(...)` before returning its (non-streamed) JSON response — the same "mint a run so the span has somewhere to land" discipline as the interactive/fire-and-watch flows, just without any SSE wrapper. Routes that touch **no** engine state (`GET /api/models`, `GET /api/mcp`, `GET /api/memory/spaces`) skip this — there's no span to place.
- **D9 — `RunKind` gains `Build` and `Pull`; `deriveRunKind`/`RUN_ROOT_NAMES` gain matching root-span names.** `src/run/run-dto.ts`'s `RUN_ROOT_NAMES` set (currently `agent.run`/`crew.run`/`workflow.run`) and `deriveRunKind()` both need `agent.build`/`crew.build` → `RunKind.Build` and `model.pull` → `RunKind.Pull`. This is the same function both `mapRunToDto` and `summarizeRunListItem` already share (Phase 3's "cannot drift apart" invariant) — one change, both projections agree. MCP test-mount / memory ingest / memory recall spans are **not** given a dedicated `RunKind` this phase (no root-span name is reserved for them) — they surface in the Runs list under the `Chat` fallback; a finer split is a natural, non-blocking follow-on once a use case needs to filter on it.
- **D10 — MCP web-mount closes a real gap, not just exposes one.** `mountAll`'s `ConsentDeps.ask` (`src/mcp/mount.ts:98-114`) today defaults to `askYesNo` over `stdinInput()` gated by `interactiveTTY()` — on the server (never a TTY), unapproved entries are **silently skipped** (`interactiveTTY()` is false, so `consent.isTTY` is false, so any entry needing fresh consent is skipped with `reason: 'consent not granted'`). Wiring `ConsentDeps.ask` to the `ConfirmPort` (D4) is what makes mounting a never-before-approved MCP server actually *possible* from the web UI — today it silently never happens.
- **D11 — Build order ships in 6 reviewable increments** (§5), same discipline as Phase 4's 9-task sequence: foundation first, then one engine seam per increment, docs/live-verify last.

## 4. Architecture

### 4.1 Contracts (`src/contracts/`, isomorphic, zod-only)

- **enums.ts:** mirror `VerifiedLevel` and `ReuseKind` (`src/verified-build/types.ts:1-11`) with parity tests (the `DegradeKind`/`StepKind`/`CrewProcess` precedent); add `RunKind.Build`, `RunKind.Pull`.
- **events.ts:** **no net-new `StatusEvent` member this phase.** The pull's live progress does **not** need a `data-download-progress` event — `SpanDtoSchema` already carries `attributes: z.record(z.string(), z.unknown())` (`src/contracts/dto.ts:62`), so each `model.pull.progress` tick span (§7.2) surfaces `DownloadProgress`'s wire-safe fields (`percent`, `bytesCompleted`, `bytesTotal`, `speedBytesPerSec`, `phase` — from `src/provisioning/types.ts:14-22`) as span attributes, read by the Models tab through the **existing** `/api/runs/:id/stream` `SpanDTO` fold. Adding a new event would have forced the runs-stream handler to carry non-span frames — new stream code, contradicting locked fork 1. MCP test-mount status reuses `McpMountEventSchema` (`data-mcp-mount`, already shipped). Builder verify-gate narration reuses `text-delta` parts (§7.1). Net-new wire events: **zero.**
- **dto.ts:**
  - `AgentProposalDTO` — near-identity of `AgentProposal` (D5): `{ name, description, systemPrompt, modelReq, suggestedServers: { packName, scopeToAgent }[], rationale }`.
  - `CrewProposalDTO` / `WorkflowProposalDTO` — near-identity of `CrewIR`/`WorkflowIR` (D5), re-validated through a contracts-owned schema (isomorphic rule: contracts cannot import `src/crew-builder`).
  - `BuildResultDTO` — `{ kind: 'written'|'declined'|'invalid'|'abandoned'|'reused'|'failed-verification', name?, files?, level?: VerifiedLevel, issues?, reason?, similarity?, stage?, detail? }` — a flattened union mirroring `BuildResult`/`CrewBuildResult` (`src/agent-builder/types.ts:22-38`, `src/crew-builder/types.ts:13-31`).
  - `ModelInventoryDTO` — `{ runtime: RuntimeKind, model: string, installed: boolean, fits: boolean, sizeBytes?, shortfallBytes? }[]`, projected from `buildRegistry()`'s merged `ModelDeclaration[]` (`src/discovery/build-registry.ts`) plus a live `fitAndRank`/`checkDiskSpace` pass (`src/provisioning/fit.ts`, `supervisor.ts`) for the pullable candidates.
  - `MemorySpaceDTO` — `{ name, chunkCount }[]`, projected from `store.stats(): Record<string, number>` (`src/memory/store.ts:178-183`).
  - `RetrievalResultDTO` — `{ id, source, text, score }[]`, projected from `RetrievalResult[]` (`src/memory/types.ts`).
  - `McpServerDTO` — `{ name, kind: McpTransportKind, agents?: string[], authKind: McpAuthKind, status: 'mounted'|'skipped'|'dormant', reason? }`, projected from `McpConfig.entries` (`src/mcp/types.ts:70-76`) joined with a `MountedRegistry.mounted`/`.skipped` snapshot (§4.3).
- **requests.ts:**
  - `BuilderBuildRequestSchema` — `{ kind: 'agent'|'crew'|'workflow', need: z.string().max(...), autoYes?: boolean, force?: boolean }`.
  - `ModelPullRequestSchema` — `{ runtime: z.enum(RuntimeKind), modelRef: z.string() }`.
  - `MemoryRecallRequestSchema` — `{ query: z.string(), space?: z.string(), topK?: z.number() }`.
  - `McpAddRequestSchema` — `{ name: z.string(), server: z.record(...) }` (the raw `mcpServers.<name>` value, mirroring `PackEntry.server`).
  - `RunLaunchResponseSchema` is reused verbatim (Phase 4) for both the builder-build "runId is carried on `data-run-start`" case and the pull-launch `{ runId }` response.
- Parity tests: `tests/contracts/verified-level-parity.test.ts`, `reuse-kind-parity.test.ts` (mirror the existing `step-kind-parity.test.ts`/`crew-process-parity.test.ts` pattern exactly).

### 4.2 Server (`src/server/builders/`, `src/server/models/`, `src/server/memory/`, `src/server/mcp/`, wired in `src/server/app.ts`)

All routes are automatically perimeter-gated (`enforcePerimeter` + `createTokenGuard`) — nothing new to add there; `handleApi`'s route-ordering discipline (sub-path matches before bare `:id`) extends to the new `/build`, `/pull`, `/recall`, `/ingest`, `/test-mount`, `/add` sub-paths, same as Phase 4's `/run` lesson.

1. **`POST /api/builders/build`** → SSE (`createUIMessageStream`/`writer.merge`/`createUIMessageStreamResponse`, exactly `handleChat`'s shape). Parses `BuilderBuildRequestSchema`, mints `runId = newRunId()`, `createRun(runsRoot, runId)`, opens a lightweight run-telemetry scope (a new small helper — §4.3 — NOT `withMcpRun`, since neither `buildAgent` nor `buildCrewOrWorkflow` mounts MCP tools at dry-run time; `agentFromProposal` in `src/agent-builder/builder.ts:104-114` explicitly builds `tools: {}`), writes `data-run-start` with the runId, dispatches on `kind` to `buildAgent`/`buildCrewOrWorkflow` with `deps.confirm`/`deps.verify.confirmReuse` bridged to `ConsentRegistry.port` (D4) and `deps.log` bridged to text-delta parts on the same writer (narration — see §7.1), writes the terminal `BuildResultDTO` as a one-shot data/text part, writes `data-run-end`. The run-telemetry scope is the **existing** `withRunTelemetry` (`src/cli/with-run.ts` — createRun + run-scoped provider + `withRunContext` + flush, no MCP mount, built for exactly the builders), invoked with a `newRunId()` instead of the CLI's pid-based id — **not** a new helper and **not** `withMcpRun` (neither `buildAgent` nor `buildCrewOrWorkflow` mounts MCP at dry-run time; `agentFromProposal` in `src/agent-builder/builder.ts:104-114` builds `tools: {}`).
2. **`GET /api/builders/agents` / `GET /api/builders/crews`** → registry lists (`agentNames()`, `Object.keys(CREWS)`/`Object.keys(WORKFLOWS)`) — thin, mirrors Phase 4's crew/workflow list handlers.
3. **`GET /api/models`** → `ModelInventoryDTO[]`, composing `buildRegistry()` (installed) with `fitAndRank` over the catalog sources (pullable, size/fit/shortfall) — same discovery pipeline `runProvision` already uses (`src/provisioning/provisioner.ts:66-91`), read-only here (no download side-effect).
4. **`POST /api/models/pull`** → fire-and-watch (D2): parse `ModelPullRequestSchema`, mint `runId`, `createRun`, `void deps.runModelPull({ runtime, modelRef, runId }).catch(→error.json)`, return `RunLaunchResponseSchema.parse({ runId })` — the exact shape of `handleCrewRun` (`src/server/crews/run.ts:41-68`). `runModelPull`'s real implementation calls `providerFor(kind).download(modelRef, { onProgress, signal, destDir })` directly (the same call `provisioner.ts`'s `downloadOne` makes) — **not** the full `runProvision` orchestration, since model selection + disk-shortfall consent are already resolved in the request (D2/locked fork 1).
5. **`GET /api/memory/spaces`** → `MemorySpaceDTO[]` from `store.stats()`. **`POST /api/memory/:space/recall`** → `RetrievalResultDTO[]` from `store.recall(query, opts)`. **`POST /api/memory/:space/ingest`** → resolves an uploaded file id through `confineToDir(fileId, deps.uploadsDir)` (the exact Phase-2 read-side pattern `handleChat` uses for image uploads, `src/server/chat/handler.ts:61-74`) then calls `store.ingest(path, { space, at: Date.now() })`.
6. **`GET /api/mcp`** → `McpServerDTO[]`, joining `loadMcpConfig()`'s `entries`/`dormant`/`warnings` with an addressable **mount-status snapshot** (new — today `mountAll`'s `mounted`/`skipped` result is per-run-only, never persisted or queryable outside the process that mounted it; the server keeps a small in-memory snapshot, refreshed on every mount attempt, analogous to the Phase-3 mtime summary cache but keyed by server name, not mtime). **`POST /api/mcp/add`** → validates `McpAddRequestSchema`, writes the entry into `mcp.json` (reusing `expandVars`-safe raw storage, `src/mcp/config.ts`), returns the updated `McpServerDTO`. **`POST /api/mcp/test-mount`** → SSE (same interactive shape as builders): mints a run, mounts ONE server via `mountAll`'s per-entry path with `ConsentDeps.ask` bridged to `ConfirmPort` (D10), streams `data-mcp-mount` events (reusing the existing schema) as consent/mount/OAuth-pop/tool-list progress, ends with the tool-list result.
7. **New `ServerDeps` fields:** `runBuilderTurn`, `runModelPull`, `memoryStore`, `mcpConfigPath`/`mcpConfigRoot` — wired in `src/server/main.ts` (mirroring how `runCrewTurn`/`runWorkflowTurn` were added in Phase 4), factories added to `src/server/launch-turns.ts`, all injectable/testable so route handlers stay unit-testable without booting Ollama/MCP (the Phase-2/3/4 "thin BFF, real impl behind an injected function" discipline).

### 4.3 Engine seams (thin adapters — no engine rewrites)

- **`ProvisionUi` web adapter** — implements `{ askYesNo, selectModels, bar }` (`src/provisioning/provisioner.ts:24-31`) against the SSE writer + `ConfirmPort`, for the one path that still needs it (none in this phase's locked scope — Model Pull bypasses `runProvision` per §4.2 item 4 — but the type is documented here since `ModelInventoryDTO`'s fit/shortfall computation reuses `fitAndRank`/`checkDiskSpace` from the same module; a future in-place "install everything recommended" flow would reuse this adapter directly).
- **Pull→spans bridge** (detailed in §7.2) — wraps `DownloadProvider.download`'s `onProgress: (p: DownloadProgress) => void` callback (`src/provisioning/types.ts:40-54`) so each callback opens+closes its own child span under a long-lived `model.pull` root.
- **Builder confirm/log adapter** — `BuilderDeps.confirm` / `BuilderVerifyDeps.confirmReuse` (`src/agent-builder/types.ts:79,86`) and `CrewBuilderDeps.confirm` / `CrewBuilderVerifyDeps.confirmReuse` (`src/crew-builder/types.ts:79,70`) → `ConsentRegistry.port` (D4); `BuilderDeps.log?`/`CrewBuilderDeps.log?` → text-delta parts on the same SSE writer (narration, §7.1).
- **Reuse the existing `withRunTelemetry`** (no MCP mount) — `src/cli/with-run.ts` **already** composes `createRun` + `initRunTelemetry` + `withRunContext` + `tel.shutdown()` teardown *without* the `mountAll` block (its doc comment names "the builders + archive" as its reason to exist). Builders, model-pull, and the plain-REST memory/mcp routes that mint an ephemeral run per D8 all reuse it verbatim — the only server-side difference from its CLI callers is the `runId` source (`newRunId()` vs the CLI's `agent-builder-${pid}`). No new helper is written. (`withMcpRun`, `src/cli/with-mcp-run.ts:77-127`, is the MCP-mounting sibling — deliberately **not** used here, since no Phase-5 flow mounts MCP at build/pull time.) One reconciliation the plan must make: the fire-and-watch pull route pre-creates the run dir (`createRun`) *before* returning `{runId}` so `/stream` never 404s, and `withRunTelemetry` calls `createRun` again inside the detached body — `createRun`'s `mkdir` is recursive/idempotent, so the double-create is safe, but the plan should confirm (or have the route pass the handle through).
- **MCP web-mount** — wires `mountAll`'s `ConsentDeps.ask` (`src/mcp/mount.ts:98-114`) to the `ConfirmPort` (closing the "server never has a TTY, so consent silently fails" gap, D10); builds the addressable mount-status snapshot (§4.2 item 6).
- **Memory** — the server handler wraps `createMemoryStore().{stats,recall,ingest}` (`src/memory/store.ts:29-189`) directly; no adapter needed beyond the D8 run-minting wrapper.

### 4.4 Web (`web/`, heavy reuse of the Phase 2–4 primitives)

- Reuse verbatim: `apiFetch` (`web/src/shared/contract/client.ts`), `createSseTransport().{stream,respond}` (`web/src/shared/transport/sse-adapter.ts`), `<DagView>` + `DagModel`/`DagStatus` (`web/src/shared/dag/`), `<ConfirmPrompt>`/`PendingConfirm` (`web/src/features/chat/confirm-prompt.tsx`, `use-status-events.ts`), `<Button>`/`RegionErrorBoundary`, `tokens.css`, and the Phase-4 crews-list shape (`web/src/features/crews/index.tsx`) as the template for all three Library sub-lists.
- **Net-new — a builder-scoped event-folding hook** (`web/src/features/builders/use-build-events.ts`): unlike `useStatusEvents` (which rides `useChat({ onData })`) or `useRunTrace` (which folds only `SpanDTO` from a run-detail stream), builders open their OWN SSE connection (`createSseTransport().stream()`, no `useChat`) whose frames mix `data-confirm` asks, text-narration deltas, and the terminal `BuildResultDTO` — the hook's pure `fold` switches on frame shape, exposing `{ narration: string[], pendingConfirm?, result? }`.
- **The guided-flow wizard** (`web/src/features/builders/agent-wizard.tsx`, `crew-wizard.tsx`): need-textarea → POST the SSE build → live narration + `<ConfirmPrompt>` when a confirm arrives (answered via `createSseTransport().respond(runId, { promptId, value })`, the existing Phase-2 respond path) → `<DagView>` proposal preview (D6) → terminal `BuildResultDTO` card.
- **The Library shell** (`web/src/features/library/index.tsx`): 3-tab shell (Models · Memory · MCP), each tab a small list/table panel — **duplicated a third time rather than prematurely abstracted** into a shared facet component (matches the repo's "don't abstract on the second use" discipline already applied to the crews/workflows lists).
- **Models tab** — inventory table + a per-row "Pull" action that POSTs `/api/models/pull`, then opens the **existing** `/api/runs/:runId/stream` and renders a progress bar off the `model.pull` span tail (reusing `useRunTrace`'s `SpanDTO` fold — no new web-side transport code, per D2).

## 7. The hard part

### 7.1 The builder streaming+confirm round-trip

A build must **stream progress AND await interactive consent mid-flow**, on one open connection, without the AI-SDK stream machinery or the confirm channel deadlocking each other.

**The concrete mechanism.** `handleChat`'s `execute: async ({ writer }) => { ... }` callback (`src/server/chat/handler.ts:94-134`) is NOT awaited by `createUIMessageStream` itself — the Response returns as soon as the stream object exists, and `execute` keeps running in the background, writing to `writer` for as long as it likes. The builder-build route uses the identical shape: `execute` calls `buildAgent`/`buildCrewOrWorkflow` and simply **doesn't return** until the whole build (including any awaited confirm) is done. Two things must be threaded correctly into that one long-lived `execute` call:
- **`deps.confirm`/`deps.verify.confirmReuse`** become `async (text) => Boolean(await deps.consent.port({ kind, question: text }, events))`, where `events` is the same `EventSink` that also carries narration (so the confirm ask rides the SAME writer as everything else — one connection, not two). `ConsentRegistry.port` (`src/server/consent/registry.ts:33-44`) returns a Promise that only settles when `POST /api/runs/:id/respond` calls `deps.consent.resolve(promptId, value)` (`src/server/consent/respond.ts:26`) — a **second, independent HTTP request** the browser fires after the user clicks Approve/Decline in `<ConfirmPrompt>`. The build's `await deps.confirm(...)` genuinely suspends `execute` until that second request lands; nothing in `createUIMessageStream` times this out, so the first (SSE) connection must be held open for as long as the human takes to answer — bounded only by `req.signal` (client navigates away / closes tab → abort) and, per the repo's reliability posture, a wall-clock cap the plan should apply around the `deps.confirm` await (mirroring `dryRunMs()`'s `withWallClock` pattern already used for the dry-run/eval calls inside `verifyAndCommitProposal`, `src/agent-builder/builder.ts:186-204`) so an abandoned wizard doesn't leak a pending build forever.
- **Narration, not a new structured event, carries verify-gate progress.** `withAgentBuildSpan`/`withBuildVerifySpan`'s `rec.event(...)` calls (`generated`, `suggested`, `validated`, `retrying`, `consent`, `structural`, `dry_run`, `golden_eval`, `gate_result`) record OTel span **events**, which — per D7/§7.2's finding — only reach `spans.jsonl` when their span closes, i.e. at the very end of the build. Tailing `/api/runs/:runId/stream` therefore shows **nothing** until the build is already over (unlike a crew/workflow run, whose child `withStepSpan`s close incrementally). Live progress the user actually watches must instead ride `BuilderDeps.log?`/`CrewBuilderDeps.log?` — already a plain `(m: string) => void` narration hook the CLI path uses (`deps.log?.(...)` calls in `src/agent-builder/builder.ts:295,414,481`) — wired to `writer.write({ type: 'text-delta', id, delta: m })` on the SAME connection. This is why the design's "these flows ALSO write spans so they appear in the Runs browser afterward" is literal: the spans give a faithful **post-hoc** trace; the narration text is what's live.
- **Requirements the review must adversarially verify:** (a) the confirm await genuinely suspends `execute` without blocking the event loop or timing out the HTTP response; (b) a client abort (`req.signal`) during a pending confirm cleans up the registry entry (or at minimum never resolves a stale build against a later, unrelated confirm answer — `promptId` unguessability already prevents cross-talk, but an abandoned build's `pendingResolvers` entry should not leak forever — a wall-clock cap resolves this); (c) the terminal `BuildResultDTO` is written exactly once, after every `rec.outcome(...)` path (`written`/`declined`/`invalid`/`reused`/`failed-verification`) — mirroring `handleChat`'s `if (result.kind !== 'answer')` one-shot-text-part discipline; (d) the run's `agent.build`/`crew.build` span closes (and thus becomes visible) even when the client disconnects mid-build — the build itself is NOT detached (unlike fire-and-watch), so an aborted `req.signal` must still let the in-flight build finish and its span close normally, not get torn down mid-stage.

### 7.2 The pull→spans bridge

`DownloadProvider.download`'s `onProgress` callback (`src/provisioning/types.ts:40-54`, driven by `ProgressTracker.update()`, `src/provisioning/progress-tracker.ts`) fires many times per download with monotonic percent + EWMA speed. The existing live-watch machinery (`handleRunStream`, `src/server/runs/stream.ts`) only ever shows spans that have **already been appended** to `spans.jsonl` by `mapRunToDto`'s next poll (every `pollMs`, default 250ms) — and `JsonlFileExporter.export()` (`src/telemetry/jsonl-exporter.ts:58-71`) is only called by the OTel SDK when a span **ends**. A naive `withProvisionSpan`-style single root span wrapping the whole download (as `runProvision` does today for the CLI, `src/provisioning/provisioner.ts:128-188`) would therefore show **nothing** in the browser until the download is completely finished or failed — no bar, just a sudden final state. That defeats the whole point of a progress bar.

**The fix — one short-lived child span per progress tick, nested under a long-lived root.** `withModelPullSpan` opens a `model.pull` root span (`tracer().startActiveSpan('model.pull', ...)`, the same `inSpan` primitive `withProvisionSpan`/`withAgentBuildSpan` already use) that stays open for the full download. Every `onProgress(p: DownloadProgress)` callback fires a **nested** `inSpan('model.pull.progress', async (span) => { span.setAttribute(...); })` call (new small ATTR keys mirroring the `MODEL_*` family already in `spans.ts` — an implementation detail for the plan) that starts and ends **immediately**, inheriting `model.pull` as its parent via the same OTel active-context propagation `withStepSpan` already relies on when nested inside `crew.run`/`workflow.run`. Each tick span closes and is appended to `spans.jsonl` right away — the poll-based tail picks it up on its next cycle, exactly like a completed workflow step.

This composes with existing mapper behavior **without any mapper change for the live-tick case**: `src/run/run-dto.ts`'s own comment on `runRootSummary` (lines 96–102) already documents that "an orphan child (its parent span never written…) is a sibling root" — precisely what a `model.pull.progress` tick span is, for as long as the `model.pull` root itself hasn't closed (and thus hasn't been exported/isn't in `buildTree`'s span set yet). `flatten()` walks every root the tree finds, orphans included, so each tick shows up as a `SpanDTO` in the live stream the moment it closes — with zero mapper changes. The **one** mapper change genuinely needed is D9: adding `'model.pull'` to `RUN_ROOT_NAMES` and a `RunKind.Pull` branch in `deriveRunKind()`, so that once the root **does** close, `runRootSummary` recognizes it, `lifecycle` flips from `Running` to `Done`/`Failed`, and `handleRunStream`'s stop condition (`dto.lifecycle !== RunLifecycle.Running`) actually fires — otherwise the SSE stream would poll forever past `maxWaitMs` even though the download is long done.

**Requirements the review must adversarially verify:** (a) tick spans are genuinely short-lived (open+close synchronously within the callback, never left dangling if `onProgress` fires again before the previous tick's `span.end()` resolves — `inSpan`'s `finally { span.end() }` makes this safe even under concurrent/rapid callbacks); (b) the `model.pull` root's status/outcome is set correctly on both the resolve and reject paths of `provider.download(...)` (mirroring `provisioner.ts`'s `downloadOne`'s try/catch, `result.failed.push(...)` on throw) so a failed pull reads `RunLifecycle.Failed`, not silently `Running` forever; (c) `RUN_ROOT_NAMES`/`deriveRunKind` are updated **together** (Phase 3's "cannot drift apart" invariant — both `mapRunToDto` and `summarizeRunListItem` call the same `runRootSummary`/`deriveRunKind`, so one edit fixes both list and detail); (d) a client that opens `/api/runs/:runId/stream` **before** the first tick span has even been written degrades cleanly (the existing "no spans yet → poll again" behavior already handles this — no new degrade path needed, just confirm it).
