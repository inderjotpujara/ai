## 1. Thesis

Phases 1–5 gave the web UI a real streaming chat surface, a Runs history/trace browser, Crews/Workflows browse-run-watch, and agent/crew builders + a Models/Memory/MCP library — but every one of those phases left chat itself **stateless**. `ChatRequestSchema.sessionId` has existed since Phase 2 (`src/contracts/requests.ts:37`) and done nothing; the browser resends the full message history on every turn and `buildTaskFromMessages` (`src/server/chat/task.ts:39-56`) collapses it into one task string server-side. Close the tab and the conversation is gone. There is no session/conversation store anywhere in the repo, and memory recall — fully built in Phase-prior slices for RAG ingest/recall — has never been wired into a chat turn. Phase 6 closes both gaps and ships the two remaining conversation basics (Sessions UI, long-run notifications) the parent spec's D18 called out as must-haves.

Four deliverables ship together as **one mega phase** (locked with the user 2026-07-16), built in 6 reviewable increments (§5):

1. **`SessionStore`** — a new `src/session/` SQLite module giving chat a real, reloadable transcript.
2. **Recall wired into chat + auto-ingest** — the existing recall pipeline (embed → dense vector search → optional rerank → budget-fit, `src/memory/retrieve.ts:48-108`) gets its first chat caller, and every completed turn writes itself back into memory so a *later* session can recall an *earlier* one.
3. **Sessions UI** — `/sessions` list + `/sessions/$id` detail, replacing the `SessionsSidebar` stub (`web/src/features/sessions/index.tsx`, currently 15 lines of placeholder text) with real history, search, rename/delete, and Markdown export.
4. **Long-run completion notifications** — a client-side poll over the *already-shipped* `GET /api/runs`, with zero new server surface.

Like every prior 30b phase, this is a **partial-slice landing**: README/ROADMAP mark Phase 6 landed, but the 30b capability itself stays 🟡 until Phase 7 (voice) and Phase 8 (polish/a11y/live-verify) close.

## 2. Scope (locked with the user 2026-07-16)

| Decision | Choice |
|---|---|
| SessionStore | New `src/session/` module, SQLite (`sessions.db`, WAL, the existing `migrate()` runner). Two tables: `sessions` (id, title, owner='local', createdAt, updatedAt, lastMessageAt, runId?) and `messages` (id, sessionId, parentMessageId [nullable, reserved], role, parts [JSON], createdAt, degraded?). |
| Session id | **Client-minted, permanent**, via `crypto.randomUUID()` on "new chat"; sent on turn 1; server does an **idempotent create-if-absent upsert**, never mints one itself. Server validates the id's format before using it as a SQLite key. Client persists the active id to `localStorage`; on reload it rehydrates via `GET /api/sessions/:id`. |
| Turn-boundary persistence | **User message persists at turn START; assistant message persists at turn COMPLETION.** A dropped connection leaves the assistant turn absent (never partial) — the user regenerates. |
| Recall | **READ + AUTO-INGEST.** Read: `injectRecall` (`src/memory/recall-tool.ts:46-62`, built, never called) wired one line into `runChatSession`. Write: fire-and-forget ingest of each completed turn into a dedicated `chat` memory space, namespaced by sessionId. Recall is **space-wide** — a new session recalls turns from every prior session. |
| Sessions UI | `/sessions` (search + cursor pagination) + `/sessions/$id` (rehydrated transcript); real `SessionsSidebar`; rename/delete/search; NAV entry; ⌘K jump-to-session. |
| Export | **Server-assembled Markdown**, `GET /api/sessions/:id/export`, from the full persisted transcript (not client-loaded, so it can't be truncated). Client downloads via a new Blob + `<a download>` helper. JSON export is a documented forward-item. |
| Notifications | **Client-side poll** over the existing `GET /api/runs` (~5s). Baselines the seen-set on the first poll (no history spam); notifies on Running→terminal transitions for long-running kinds (Crew/Workflow/Agent) past a duration threshold. No new server endpoint, no event bus. In-app toast + optional browser `Notification` API + a Settings toggle; backs off when the tab is hidden. |

**Explicitly out of scope (deferred, none blocking):** `parentMessageId` branch/fork threading; JSON export; server-push notifications/global SSE bus; a real ANN index + hybrid FTS fusion for memory (pre-existing gap, unaffected by this phase); session retention/GC; edit-in-place history rewrite.

## 3. Design decisions (D-series)

- **D1 — `SessionStore` mirrors `createMemoryStore`'s factory-returns-closure shape and reuses two existing primitives verbatim.** `createSessionStore(config: {path?: string}, deps: {})` (`src/session/store.ts`, new) opens `sessions.db` with the identical WAL/`busy_timeout`/`foreign_keys` pragma trio `SqliteStore`'s constructor already runs (`src/memory/sqlite-store.ts:38-41`) and calls the same `migrate(db, migrations)` runner (`src/db/migrate.ts:6-20`) — one migration, `'init-sessions-and-messages'`. `deps` stays an empty object for now (no embedder analog needed), kept as a second arg only for signature parity with `createMemoryStore` (`src/memory/store.ts:29`) and as a future test seam (e.g. a clock override). New config entry `AGENT_SESSIONS_PATH` (string, default `'sessions'`), following `config/schema.ts`'s exact `AGENT_MEMORY_PATH` precedent (`src/config/schema.ts:128-131`).
- **D2 — Session id is client-minted and format-validated at the zod layer, not a new code path.** `ChatRequestSchema.sessionId` (`src/contracts/requests.ts:37`, already optional) gains a regex constraint matching `crypto.randomUUID()`'s v4-UUID shape. A malformed id is rejected by the SAME `ChatRequestSchema.parse(await req.json())` call `handleChat` already makes (`src/server/chat/handler.ts:40`) — no separate validation branch. The server does `INSERT OR IGNORE` (create-if-absent); it never generates an id itself, matching the locked "no server-mint, no data-session round-trip" requirement.
- **D3 — Turn-boundary persistence reuses the exact value `handleChat` already branches on — no stream tap.** The user's latest message persists right after `ChatRequestSchema.parse` succeeds (`handler.ts:40-45`), before `deps.runChatTurn(...)` dispatches. The assistant message persists after `deps.runChatTurn(...)` resolves (`handler.ts:106`) and reuses `result` — for `kind === 'answer'`, `OrchestratorResult.text` (`src/core/orchestrator.ts:22`) already holds the complete final text (it streamed via `streamSink` too, but the value is still available on the resolved object); for `'gap'`/`'resource'`, `result.message` is exactly what `handler.ts:120-125` already writes as the one-shot text part. No extra `ReadableStream` tap is needed — the deferred chunk-counting note at `handler.ts:98-103` is unrelated and stays deferred.
- **D4 — Persistence is append-only and idempotent, keyed by the client-generated message id, and persists ONLY the latest turn.** `UiMessageLikeSchema.id` (`requests.ts:29`) already exists per message; `appendMessage` uses `INSERT OR IGNORE` on it, so a retried/duplicate POST is a safe no-op. Critically, only the **latest** user message is persisted per turn — the same one `buildTaskFromMessages`'s `lastUserIdx` already isolates (`src/server/chat/task.ts:40-42`) — never the full `body.messages` array, because the browser resends the ENTIRE history every request (`task.ts`'s own doc comment, lines 31-38); blindly persisting the whole array on every turn would re-insert every prior turn, every time.
- **D5 — Recall (read) is engine-level and shared by CLI+server; auto-ingest (write) is server-only.** `ChatSessionDeps` (`src/cli/run-chat-session.ts:19-29`) gains an optional `memoryStore?: MemoryStore` field. `runChatSession` (`run-chat-session.ts:60-105`) adds exactly one call right after the existing media-ingest block resolves (currently ends at line 82, `task = ingested.prompt;`): `if (deps.memoryStore) task = await injectRecall(deps.memoryStore, { space: CHAT_MEMORY_SPACE }, task);`. Because recall is space-wide (no namespace filter — the whole point of "self-remembering across sessions"), both CLI and server callers get the identical benefit uniformly, matching the shared-engine-seam discipline the parent spec's D13 already established. Auto-ingest, by contrast, needs a `sessionId` to scope its namespace — a concept only the server's chat-persistence layer has — so it lives in `src/server/chat/handler.ts`, not inside `runChatSession`. The CLI's `chat` command still gains the READ half by wiring its own `memoryStore` via `src/cli/memory.ts`'s existing `makeRealStore` helper (`src/cli/memory.ts:48`); it never writes to the `chat` space (no session to namespace by).
- **D6 — Auto-ingest reuses the store's existing dedup primitive, extended to raw text.** `MemoryStore.ingest()` (`src/memory/store.ts:116-137`) already has the exact collision-safe semantics auto-ingest needs — `SqliteStore.seenDoc`/`recordDoc` (`src/memory/sqlite-store.ts:80-105`) — but is shaped around a file path. `createMemoryStore` gains one small additive method, `rememberOnce(text, { space, namespace, source, at })`, that runs the identical seen/record guard against a caller-supplied `source` id and a `sha256` of `text` directly (no file, no path resolution) — reusing the `documents` table's existing columns (`space, source, hash, chunks, at`, `sqlite-store.ts:14-16`) as-is, no schema change. `source = `chat:${sessionId}:${assistantMessageId}`` — stable per turn, so a duplicate completed-turn write is a no-op for free.
- **D7 — The persisted `degraded` marker reuses the existing `EventSink` tap, not a new engine query.** `handleChat`'s `events` closure already fires on every `StatusEvent` including `StatusEventType.Degrade` (`handler.ts:96-103`); the closure gains one boolean (`degradedThisTurn ||= true` on that case), stamped onto the persisted assistant row's `degraded` column — reusing `ChatMessageDtoSchema`'s already-reserved `degraded` field (`src/contracts/dto.ts:124-131`) instead of inventing a new one.
- **D8 — Rehydrate reuses `ChatMessageDtoSchema`; export reads the raw stored parts, not the wire DTO.** The new `SessionDtoSchema` embeds `messages: ChatMessageDTO[]` (reusing `dto.ts:124-131` verbatim) for the client's reload path — enough for the client to synthesize UIMessage-shaped objects (`{id, role, parts:[{type:'text', text}]}`) the same way `MessageList`/`joinedText` already expect (`web/src/features/chat/index.tsx:17-22`). Markdown export, by contrast, is assembled server-side straight from the `messages` table's raw `parts` JSON — a one-shot server render, not a live wire contract — so it needs no additional DTO.
- **D9 — Export must be fetched, not linked, because of the existing token perimeter.** Every `/api` call requires `Authorization: Bearer <token>` (`web/src/shared/contract/client.ts:14-17,30-32`); a plain `<a href="/api/sessions/:id/export">` would 401 on a bare navigation (no header). The client instead performs a `fetch()` carrying the header, reads the Markdown body as text, and constructs a `Blob` + object URL for a synthetic `<a download>` click — the "no precedent in `web/`" helper the locked scope calls for. This is a direct, mechanical consequence of the parent spec's D17 perimeter design, not a new security decision.
- **D10 — Sessions pagination mirrors Runs' cursor exactly, not Memory's bare-array idiom.** `GET /api/sessions` returns `{items, nextCursor?, total}` (`SessionListResponseSchema`), reusing the identical `base64url(sortKey:id)` opaque-cursor contract already implemented for `GET /api/runs` (`encodeCursor`/`decodeCursorId`, `src/server/runs/list.ts:22-33`). Sort key is `lastMessageAt` (falling back to `createdAt` for a session with no messages yet), descending, id tie-break. Because sessions live in a real SQLite table (not a filesystem `readdir`), the query does the sort/filter/keyset-page in SQL rather than `list.ts`'s in-process array approach — only the internals differ; the wire shape and the cursor's opaque contract with the client are identical.
- **D11 — Notifications are baseline-then-diff over the existing run list; no new endpoint, no missed-transition edge case.** Run completion in this codebase is entirely disk-derived — `runRootSummary` (`src/run/run-dto.ts:111-141`) re-reads `spans.jsonl` on every request/poll to decide `RunLifecycle`; there is no completion event bus anywhere in the server, and inventing one purely to notify the browser would be new server-side machinery for a UI-only convenience. So an `AppShell`-level hook instead polls `GET /api/runs` (already shipped) on `AGENT_WEB_NOTIFY_POLL_MS` (new config entry, default `5_000`), filtering to `kind ∈ {Crew, Workflow, Agent}` (`RunKind`, `src/contracts/enums.ts:116-125`). The FIRST successful poll only seeds a `Map<runId, RunLifecycle>` — it never notifies (baseline), which is what keeps pre-existing terminal runs from firing on mount. Every later poll diffs: a run last seen `Running` that is now `Done`/`Failed`, with `durationMs` above `AGENT_WEB_NOTIFY_MIN_DURATION_MS` (new config entry, default `60_000`), fires exactly once. §7.2 proves this scheme cannot miss a qualifying transition given the default poll/threshold ratio.
- **D12 — Regenerate/edit+resend stay linear this phase; `parentMessageId` is written as a nullable column but not consumed for threading.** Regenerate (`chat/index.tsx:101`) and edit+resend (`chat/index.tsx:43-48`, a local `setMessages` truncation) each simply produce a new assistant turn under the same `sessionId`. Because persistence is append-only (D4), a regenerated/edited-away answer stays in the database even though the live client view hides it until reload — a reload shows the COMPLETE linear history, including turns the user "edited away" client-side in the current tab. This is a deliberate, stated scope line, not a silently swept gap: true edit-in-place / branch history is exactly what `parentMessageId` (nullable now) is reserved for — Slice 41 consumes it.

### 4.1 Contracts (`src/contracts/`, isomorphic, zod-only)

- **enums.ts:** no net-new enum this phase — `ChatRole`, `RunKind`, `RunLifecycle` already cover everything a session/notification needs (mirrors Phase 5's "no net-new `StatusEventType`" callout, `dto.ts` doc precedent).
- **dto.ts:**
  - `SessionListItemDtoSchema` — `{ id, title, owner: z.string() /* reserved, 'local' */, createdAt: z.number(), updatedAt: z.number(), lastMessageAt: z.number().optional(), runId: z.string().optional() }`.
  - `SessionDtoSchema` — `SessionListItemDtoSchema`'s fields plus `{ messages: z.array(ChatMessageDtoSchema) }`, reusing `ChatMessageDtoSchema` (`dto.ts:124-131`) verbatim — no new message DTO.
- **requests.ts:**
  - `ChatRequestSchema.sessionId` (existing, `requests.ts:37`) gains a UUID-v4-shape regex constraint (matches `crypto.randomUUID()`'s output).
  - `SessionListQuerySchema` — `{ search?: z.string(), limit: z.coerce.number().int().positive().max(200).default(25), cursor?: z.string() }`, `RunListQuerySchema`'s shape (`requests.ts:65-76`) minus the outcome/degraded/kind facets that don't apply to sessions.
  - `SessionListResponseSchema` — `{ items: z.array(SessionListItemDtoSchema), nextCursor: z.string().optional(), total: z.number() }`, byte-for-byte `RunListResponseSchema`'s shape (`requests.ts:78-84`).
  - `SessionRenameRequestSchema` — `{ title: z.string().min(1).max(200) }`, bounded the same way every other free-text body is (`BuilderBuildRequestSchema.need`, etc.).
- No new parity tests: `SessionDTO`/`SessionListItemDTO` are contract-owned with no engine-side type to mirror — same reasoning `dto.ts` already gives for `RunKind`.

### 4.2 Server (`src/server/sessions/`, wired in `src/server/app.ts`)

1. **`GET /api/sessions?search=&cursor=&limit=`** → `handleSessionList`, SQL keyset page over `sessions`, projected to `SessionListResponseSchema`.
2. **`GET /api/sessions/:id`** → `handleSessionDetail`, 404 if absent, `messages` via `sessionStore.getMessages(id)`.
3. **`PATCH /api/sessions/:id`** → `handleSessionRename`, `SessionRenameRequestSchema` body.
4. **`DELETE /api/sessions/:id`** → `handleSessionDelete`, cascades `messages` in one transaction.
5. **`GET /api/sessions/:id/export`** → `handleSessionExport` — the server's **first non-JSON API response** (every existing handler returns `app.ts`'s `json()` helper, `app.ts:86-94`); this route returns `new Response(markdown, { headers: { 'content-type': 'text/markdown; charset=utf-8', ...ISOLATION_HEADERS } })`. **Route-ordering discipline applies again:** `/api/sessions/:id/export` must be matched BEFORE the bare `/api/sessions/:id` regex, the same stream-before-detail rule already governing `/api/runs/:id/stream` (`app.ts:160-179`).
6. **`POST /api/chat`** (existing route, `handler.ts`) extended per D3/D4/D5/D6/D7 — no new route, same handler.
7. **New `ServerDeps` field:** `sessionStore: SessionStore`, wired in `src/server/main.ts` alongside the existing `memoryStore` construction (`main.ts:152-161`) — same file, same "nothing lazy needed" discipline (`SqliteStore`'s constructor is cheap/synchronous, unlike `LazyEngine`'s Ollama-dependent pieces).

### 4.3 `SessionStore` engine (`src/session/`)

- **`migrations.ts`** — one migration, `'init-sessions-and-messages'`: `sessions(id PK, title, owner NOT NULL DEFAULT 'local', created_at, updated_at, last_message_at NULL, run_id NULL)`; `messages(id PK, session_id NOT NULL REFERENCES sessions(id), parent_message_id NULL, role, parts TEXT NOT NULL, created_at, degraded NULL)`.
- **`store.ts`** — `createSessionStore(config, deps)` returns: `upsertSession(id, { defaultTitle, at })` idempotent create-if-absent (title defaults from the first user message's truncated text, never overwritten by a later upsert); `getSession(id)`; `renameSession(id, title, at)`; `deleteSession(id)` (transaction: delete `messages` then the `sessions` row); `listSessions({ search?, cursor?, limit })` → SQL keyset page (D10); `appendMessage(sessionId, { id, role, parts, parentMessageId?, degraded? }, at)` idempotent (`INSERT OR IGNORE` on message id) + touches `sessions.updated_at`/`last_message_at`/`run_id`; `getMessages(sessionId)` ordered by `created_at` ascending; `close()`.
- **`MemoryStore.rememberOnce`** (D6) — an additive method on `src/memory/store.ts`'s returned closure, alongside the existing `remember`/`ingest`/`recall`.

- `src/cli/chat.ts` gains the same optional `memoryStore` wiring via `cli/memory.ts`'s `makeRealStore` (`src/cli/memory.ts:48`) — CLI gets the READ benefit (D5).
- `handler.ts` gains: (a) `sessionId` extraction + `upsertSession` right after `body = ChatRequestSchema.parse(...)` succeeds (`handler.ts:40-45`), before the media-by-reference block begins (`handler.ts:47`); (b) the user-message persist immediately after; (c) the `degradedThisTurn` tap on the existing `events` closure (`handler.ts:96-103`); (d) after `deps.runChatTurn(...)` resolves (`handler.ts:106`) and before `rec.outcome(...)` (`handler.ts:126`): the assistant-message persist, **awaited** (must be durable before the stream ends), and the `rememberOnce` auto-ingest call, **fired with `void`** (must not block the stream — D5/D6).

## 7. The hard part

### 7.1 Turn-boundary persistence + recall under streaming

A chat turn must persist the user's ask durably before any work starts, persist the assistant's answer durably only once it's actually complete, and do BOTH without letting a background memory write slow down — or corrupt — the live SSE stream.

**The concrete mechanism.** The user-message persist is awaited inside `handleChat`'s `execute` callback BEFORE `deps.runChatTurn(...)` is called — a genuinely new serialization point on a path that was previously all fire-and-stream. The assistant-message persist is awaited too, but only after `runChatTurn` resolves and reuses the SAME `result.text`/`result.message` value the handler already branches on (D3) — no extra `ReadableStream` tap. The `rememberOnce` auto-ingest call, by contrast, is fired with `void` (never awaited) — the SSE response must be free to end without waiting on an embedding round-trip, which the store's own `memory.recall`/`memory.ingest` spans already prove is not instant.

**Requirements the review must adversarially verify:**
- (a) The user-persist genuinely completes (or is at least issued and its promise held) before the first token can reach the browser — no race where a fast first token beats the write.
- (b) `req.signal` abort mid-turn does not leave persistence half-done in a way that corrupts the session: either the assistant row is written in full or not at all — never partial/malformed JSON in `parts`.
- (c) `upsertSession`'s `INSERT OR IGNORE` is genuinely a no-op on a repeat id (not a constraint-violation throw that turns a legitimate retry into a 500).
- (d) `rememberOnce`'s hash+source dedup key cannot collide across DIFFERENT turns in the same session — `source` includes the assistant message's own (unique, client-generated) id, so no two turns share a key.
- (e) **The dropped-stream case is a deliberately visible gap, not a silent loss:** if the connection dies after the user row is written but before `runChatTurn` resolves, the assistant row is simply never written. `GET /api/sessions/:id` then shows a user turn with no reply — exactly the locked "user regenerates" recovery path, not an attempt at partial-text checkpointing (which would require the extra stream tap `handler.ts:98-103` already defers for an unrelated reason).

### 7.2 Notification diff correctness

The poll-and-diff scheme must never notify about pre-existing (historical) runs, must notify exactly once per qualifying completion, and must never miss a transition that happens between two poll ticks.

**The concrete mechanism.** The hook keeps a `Map<runId, RunLifecycle>`. The FIRST successful poll after mount only seeds this map — it calls no notify sink, full stop, even if the run list already contains terminal Crew/Workflow/Agent runs from a prior session. Every subsequent poll diffs the fresh list against the map: a run last recorded `Running` that is now `Done`/`Failed`, with `durationMs` above the threshold, notifies once; the map entry is then updated to the terminal lifecycle, which makes the transition guard (`was Running && now terminal`) permanently false for that run afterward — dedup falls out of the data structure, no extra `Set` needed.

**The correctness argument (why no long-running run can be missed).** A run gated by `durationMs > AGENT_WEB_NOTIFY_MIN_DURATION_MS` (default `60_000`) cannot both start and finish inside one poll interval `AGENT_WEB_NOTIFY_POLL_MS` (default `5_000` — a 12× margin), so it is provably observed as `Running` on at least one poll before it can ever be observed as terminal. The only case where a run is seen terminal on its first-ever appearance in the map is the baseline poll itself, which is explicitly excluded from notifying. There is no missed-transition edge case left to patch around **as long as the threshold/interval ratio holds** — the review must verify that relationship is documented as an invariant (an operator setting `AGENT_WEB_NOTIFY_POLL_MS` above `AGENT_WEB_NOTIFY_MIN_DURATION_MS` would reopen it).

**Requirements the review must adversarially verify:**
- (a) The baseline poll never calls the notify sink, regardless of what it finds.
- (b) The transition check is `Running → terminal` specifically (not "terminal now"), so a run that was already terminal at baseline never fires later.
- (c) Tab-hidden backoff slows the poll cadence but never drops or resets the seen-map — a hidden tab that becomes visible again must resume from its existing map, not re-baseline (which would silently swallow any transition that happened entirely while hidden).
- (d) The `kind` filter (`Crew`/`Workflow`/`Agent` only) is applied before the transition check, so a long chat turn never fires a notification it was never meant to.
