## 1. Thesis

Slice 29 gave the CLI real speech-to-text — mic or file, `sherpa-onnx-node` + moonshine, spliced into the prompt — but explicitly scoped voice-**out**, barge-in, and true hold-to-talk to "Slice 30's browser UI, where `getUserMedia({audio:{echoCancellation:true}})` gives real AEC and `keydown`/`keyup` give a real hold-to-talk gesture, essentially for free" (`docs/architecture.md:3338-3342`). Phase 7 collects on exactly that promise, narrowly: **hands-free dictation**, nothing more. Speaking fills the composer with transcribed text, interim then final; the user still reviews it and presses Send, same as if they'd typed it. No TTS, no barge-in, no interrupting an in-flight response — this is an input modality, not a new interaction model.

**This phase overrides the parent spec's D8 engine choice.** D8 assumed sherpa-onnx WASM in the browser, mirroring the CLI's `sherpa-onnx-node`. Re-validated this session: sherpa-onnx has no first-party browser npm package in 2026 — the parent spec's own biggest flagged risk (§5 item 1: "single-threaded fallback if COOP/COEP isolation is troublesome... before the voice phase") undersold it; a working browser build means an Emscripten build-from-source plus embedding a *second* ONNX runtime alongside the one `@huggingface/transformers` already pulls in for other subsystems. Phase 7 instead uses **transformers.js** (`@huggingface/transformers`, already a root `package.json` dependency — `package.json:43` — whose `onnxruntime-web` transitive dependency the workspace already resolves, `bun.lock:146`) running Moonshine ASR + Silero VAD through HuggingFace's own `moonshine-web` reference architecture: one `AutoModel`/pipeline load for both. This eliminates the second-runtime risk entirely and reuses infrastructure already in the tree. The transcription **contract** — `VoiceFrames` (mono Float32 @ 16kHz) and the `voice.transcribe` telemetry span — carries over unchanged from Slice 29, per D8's original intent.

Like every prior 30b phase, this is a **partial-slice landing**: README/ROADMAP mark Phase 7 landed, but the 30b capability itself stays 🟡 until Phase 8 (polish/a11y/live-verify) closes.

## 2. Scope (locked with the user 2026-07-18)

| Decision | Choice |
|---|---|
| STT engine | **transformers.js + Moonshine + Silero VAD**, overriding parent D8's sherpa-WASM assumption (see Thesis). `onnx-community/moonshine-base-ONNX` (~120–150 MB) default, `onnx-community/moonshine-tiny-ONNX` (~76 MB) selectable; `onnx-community/silero-vad` (~2.2 MB) always loaded alongside. WebGPU (Chrome/Edge) with automatic WASM fallback (Safari, no WebGPU). |

| Cross-origin isolation | **Already shipped** — no new work (see D8 below). |

**Explicitly out of scope (deferred, none blocking):** TTS / voice-**out** (a future phase); barge-in / stream-interrupt / cancelling an in-flight response (this is dictation-only — the user always presses Send); server-side STT (everything in this phase is client-side, no new server route); any non-localhost/multi-user voice concern (out of this phase's — and this repo's current — perimeter entirely).

## 3. Design decisions (D-series)

- **D1 — STT engine overrides parent D8; sherpa-onnx WASM is rejected, with the reasoning recorded.** transformers.js + Moonshine + Silero VAD, loaded through ONE `AutoModel`/pipeline call (HuggingFace's own `moonshine-web` reference architecture bundles VAD gating and ASR inference together) rather than two separately-versioned engines. WebGPU is preferred when available; capability detection falls back to WASM automatically (Safari has no WebGPU as of this writing). Sizes: `moonshine-base-ONNX` ~120–150 MB (default, better accuracy), `moonshine-tiny-ONNX` ~76 MB (selectable, faster/lighter), `silero-vad` ~2.2 MB (always loaded, both tiers). All three lazy-load on first enable and persist in the browser's Cache API so a reload doesn't re-download.
- **D2 — Dictation-only interaction model; the injection point is the Composer's own text state, not the chat transport.** `Composer` (`web/src/features/chat/composer.tsx:43`) already owns `value` via `useState(initialValue)` and only hands it to the parent through `handleSubmit` → `onSend(text, uploadIds)` (`composer.tsx:78-87`), which `ChatArea` wires to `handleSend(text, uploadIds)` → `sendMessage({text}, {body})` (`web/src/features/chat/index.tsx:77-97`). Voice input writes interim/final transcript text directly into that same `value` state (via `setValue`) — the exact string the user would have typed. **Zero changes to `handleSend`, `sendMessage`, or the transport.** The mic button and its hook mount inside `Composer`, in the `composer-dropzone` section (`composer.tsx:90-96`) alongside the existing attachment UI, following the same "drop-zone owns its own affordances" pattern attachments already established there.
- **D3 — Audio capture: AEC-enabled `getUserMedia` → AudioWorklet downsample, the correctness-critical unit-test surface.** `getUserMedia({audio:{echoCancellation:true, noiseSuppression:true, autoGainControl:true}})` gives real acoustic echo cancellation for free — the exact wall Slice 29's CLI hit (self-echo, no AEC in a terminal, `docs/architecture.md:3332-3333`) simply doesn't exist in a browser tab. An `AudioWorkletProcessor` downsamples the browser's native 48kHz mono/stereo capture to 16kHz mono `Float32Array`, matching `VoiceFrames`'s fixed `sampleRate: 16000` (`src/voice/types.ts:2-5`) exactly. This resample/format math — not the UI — is the piece with a wrong-answer failure mode (garbled or silently-dropped audio) and is the primary unit-test target (§7.1).
- **D4 — Engine runs in a Web Worker; capture runs in an AudioWorklet.** Two separate off-main-thread contexts, not one: the AudioWorklet (audio-rendering thread, real-time-constrained) only ever does cheap sample-rate conversion and buffering; the Web Worker hosts transformers.js's heavier VAD+ASR inference. Frames cross from worklet → main thread → worker via `postMessage`/`Transferable` `Float32Array` buffers, keeping the main UI thread free for React rendering throughout a capture.
- **D5 — `VoiceFrames` is lifted into `src/contracts/` as a plain, non-zod shared type; `CaptureSource` is mirrored as a parity-tested enum.** CLI (`src/voice/`) and the new web voice code need the IDENTICAL `{samples: Float32Array, sampleRate: 16000}` shape, but `src/voice/` is Node-only (pulls Bun-specific spawn/ffmpeg glue and, transitively, native addon loading) and must never be imported by `web/`. The type moves to a new `src/contracts/voice.ts`, and `src/voice/types.ts:1-5` re-exports it rather than redefining it — one definition, two importers, no drift. **The tradeoff, stated plainly:** every other `src/contracts/` type is a zod schema (the "isomorphic, zod-only" convention every prior phase has followed, e.g. Phase 5 §4.1's "contracts cannot import `src/crew-builder`" rule). `VoiceFrames` is a deliberate, narrow exception — it is never serialized over HTTP (audio never leaves the browser tab in this phase; there's no wire boundary to validate), so a zod schema for a `Float32Array` field would add ceremony with no round-trip to protect. `CaptureSource` (`src/voice/types.ts:7-10`, currently `Mic`/`File`) DOES get the normal treatment — mirrored into `src/contracts/enums.ts` with a parity test (`tests/contracts/capture-source-parity.test.ts`, following the `runtime-kind-parity.test.ts`/`step-kind-parity.test.ts` precedent) — because the browser path needs it as a `voice.transcribe` span attribute value (D-telemetry, §8), which is exactly the kind of engine-enum-as-observable-value case the existing parity tests already guard.
- **D6 — Module layout: `web/src/features/voice/`.** `audio-capture.ts` (`getUserMedia` + the AudioWorklet wrapper); `stt-engine.ts` (Web Worker bootstrap + transformers.js Moonshine/VAD lazy-load, Cache-API persistence); `vad.ts` (the Silero gating/segmentation logic that decides speech-start/speech-end boundaries); `use-voice-input.ts` (the hook orchestrating both gestures, exposing `{interim, final, listening, level}` to the UI); `mic-button.tsx` (the hold/tap affordance, mounted per D2); `waveform.tsx` (a live level/waveform indicator while listening). This is the concrete implementation of the parent spec's `SttEngine`+`AudioCapture` voice port (`2026-07-08-slice-30-local-web-ui-design.md:243`, "Adapter: sherpa WASM + AudioWorklet" — the adapter name in that line is now stale per D1's override, corrected here).
- **D7 — Settings + config mirror the Phase-6 OS-notify pattern exactly.** `SettingsArea` (`web/src/features/settings/index.tsx`) gains a second control block alongside the existing OS-notify toggle: a voice-input enable toggle (`agent.voiceInputEnabled` in `localStorage`, an exported `isVoiceInputEnabled()` accessor mirroring `isOsNotifyEnabled()` at `settings/index.tsx:17-19`), a mic-permission request path, and a model-tier selector (base/tiny). Tunables (VAD sensitivity, silence-timeout ms, default model id) are **not** hardcoded client-side constants — they follow the exact `AGENT_WEB_NOTIFY_*` precedent (`src/config/schema.ts:500-511`): new `AGENT_WEB_VOICE_VAD_SILENCE_MS`, `AGENT_WEB_VOICE_DEFAULT_MODEL` config entries, injected into the served page as `window.__AGENT_VOICE_*` globals by `renderIndexHtml` (`src/server/main.ts:69-101`, which already injects `window.__AGENT_NOTIFY_POLL_MS__`/`window.__AGENT_NOTIFY_MIN_DURATION_MS__` the identical way — Phase 7 adds a third/fourth global to the same `tokenScript` string, no new mechanism).
- **D8 — Cross-origin isolation is an already-satisfied precondition, not new work.** `web/vite.config.ts:6-9,19-20` already sets `Cross-Origin-Opener-Policy: same-origin` / `Cross-Origin-Embedder-Policy: require-corp` for dev/preview, and `src/server/isolation-headers.ts` serves the identical pair in production — both put in place specifically "so the frontend can later use sherpa WASM `SharedArrayBuffer`" (`isolation-headers.ts:1-3`). `SharedArrayBuffer` availability benefits transformers.js's threaded WASM backend the same way it would have benefited sherpa; **nothing needs to change here**, only be noted as already-done groundwork this phase finally exercises.
- **D9 — Graceful degrade, never crash, matching the repo's standing reliability contract.** Mic permission denied → an explicit inline message + a disabled mic button (not a silent no-op). WebGPU absent → automatic WASM fallback (D1), invisible to the user beyond a slower first load. Model download failure → a toast + fall back to text-only input (the composer still works exactly as before voice existed). None of these paths ever throw uncaught into the chat UI.

## 4. Architecture

### 4.1 Contracts (`src/contracts/`, isomorphic)

- **`voice.ts`** (new) — the lifted `VoiceFrames` plain type (D5), re-exported by `src/voice/types.ts` for the CLI side. Not a zod schema (documented exception, D5).
- **`enums.ts`** — `CaptureSource` mirrored (`Mic`/`File`, D5) with `tests/contracts/capture-source-parity.test.ts`.
- No new DTOs, no new request/response schemas: this phase adds **zero server routes** (everything is client-side), so there is nothing new to carry over the wire.

### 4.2 Web (`web/src/features/voice/`, new — the whole build surface)

1. **`audio-capture.ts`** — wraps `getUserMedia` (D3's constraints) + registers an `AudioWorkletProcessor` that downsamples 48kHz → 16kHz mono and emits fixed-size `Float32Array` chunks via `port.postMessage` (transferable, zero-copy).
2. **`stt-engine.ts`** — boots a dedicated Web Worker (D4); on first use, lazy-loads the selected Moonshine model + Silero VAD through transformers.js's `AutoModel`/pipeline, checking the Cache API first (skip re-download); exposes a `transcribe(frames: VoiceFrames): Promise<{text, isFinal}>`-shaped message protocol to the main thread.
3. **`vad.ts`** — the Silero-driven segmentation logic: buffers incoming worklet chunks, flags speech-start/speech-end, and (tap-to-toggle mode only) auto-stops a listening session on sustained silence. Pure/testable independent of the real worker (§7.1).
4. **`use-voice-input.ts`** — the hook wiring both capture gestures (hold-to-talk via `keydown`/`keyup`+`pointerdown`/`pointerup`; tap-to-toggle via a single click + VAD auto-stop) to `audio-capture.ts` + `stt-engine.ts`, exposing `{listening, level, interim, final}` to the UI. Owns the worker lifecycle (spawn once, reuse across sessions, terminate on unmount).
5. **`mic-button.tsx`** — the composer-mounted affordance (D2); shows permission/degrade states (D9) inline.
6. **`waveform.tsx`** — a lightweight live level/waveform indicator driven by `use-voice-input`'s `level` stream while listening.

### 4.3 Composer wiring (existing file, extended not rewritten)

`composer.tsx` imports `mic-button.tsx` into its `composer-dropzone` section (`composer.tsx:90-96`, alongside the existing attachment chips), and threads `use-voice-input`'s interim/final transcript into its own `setValue` call — appending on `interim` updates, replacing/finalizing on `final`. The existing `handleSubmit`/`onSend` path (`composer.tsx:78-87`) is untouched; voice is purely another way to populate `value` before the same Send.

### 4.4 Settings + config wiring

- `settings/index.tsx` gains the toggle + mic-permission + model-tier controls (D7), mirroring the existing `notify-os-toggle` block (`settings/index.tsx:60-77`) structurally.
- `src/config/schema.ts` gains `AGENT_WEB_VOICE_VAD_SILENCE_MS` (number, sensible default) and `AGENT_WEB_VOICE_DEFAULT_MODEL` (string, default `'moonshine-base'`), following the `AGENT_WEB_NOTIFY_POLL_MS`/`AGENT_WEB_NOTIFY_MIN_DURATION_MS` doc-comment convention exactly (`schema.ts:500-511`).
- `src/server/main.ts`'s `renderIndexHtml` (`main.ts:69-101`) gains a third config plumb-through, appending `window.__AGENT_VOICE_VAD_SILENCE_MS__`/`window.__AGENT_VOICE_DEFAULT_MODEL__` onto the same `tokenScript` string that already carries the notify globals — one function, one injection point, no new mechanism.

### 4.5 Data flow: mic → capture → VAD → transcribe → composer

Hold-to-talk: `keydown`/`pointerdown` on `mic-button.tsx` → `use-voice-input` opens `getUserMedia` (if not already open) → the AudioWorklet streams 16kHz `Float32Array` chunks to the main thread → chunks buffer in the hook (no VAD gating in this mode — the key/pointer IS the segment boundary) → `keyup`/`pointerup` flushes the worklet's residual buffer, closes the segment, and hands the accumulated `VoiceFrames` to `stt-engine.ts`'s worker → the worker's Moonshine pass returns text → `composer.tsx`'s `setValue` receives it as the final transcript. Tap-to-toggle: a single tap opens the same capture path, but `vad.ts`'s Silero pass now gates segmentation — each speech-start/speech-end cycle closes and transcribes its own segment independently (so a long tap-to-toggle session can append several sentences before the user taps again to stop), with interim partial results optionally surfaced mid-segment and the final result replacing them once that segment's transcription resolves. In both modes, the transcript never touches `sendMessage`/`handleSend` (D2) — it stops at the composer's own `value`, exactly where a typed character would land.

## 7. The hard part

### 7.1 AudioWorklet capture/downsample correctness + the VAD-gated segmentation state machine

Two correctness-critical pieces, both about **never silently corrupting or dropping audio**, mirroring the exact failure class Slice 29's CLI capture already had to guard against once (`src/voice/capture.ts:12-19`'s `.bytes()` vs `.arrayBuffer()` bug — a silent all-zero buffer with no thrown error).

**The concrete mechanism.** The AudioWorklet receives the browser's native audio in fixed-size render quanta (128 frames per call, per the Web Audio spec) at whatever the device's native rate is (typically 48kHz); these must be resampled to a *continuous* 16kHz stream despite each `process()` callback boundary being arbitrary with respect to the resample ratio. Exactly like `carryPcmChunk` (`src/voice/capture.ts:45-60`) carries a 0–3-byte remainder across ffmpeg-pipe reads so no chunk boundary drops a partial sample, the AudioWorklet's downsampler must carry its own fractional-sample remainder (or filter-state, if a proper anti-aliasing resample is used) across `process()` calls — converting each render quantum independently, with no cross-call state, would either drop trailing samples at every boundary or introduce audible clicks from a naive rate-conversion that resets its phase every 128 frames.

Separately, the VAD-gated segmentation state machine (tap-to-toggle mode) must correctly track: speech-start (VAD flips silent→speech) → begin buffering a segment; speech-end (VAD flips speech→silent, sustained past the configured silence window) → close the segment, hand it to `stt-engine.ts`'s `transcribe()`, append the result to the composer, and re-arm for the next segment (a tap-to-toggle session can span multiple speech/silence cycles before the user taps to actually stop). Hold-to-talk mode has a *different* boundary condition: the segment is deliberately NOT VAD-gated — it starts on `keydown`/`pointerdown` and ends on `keyup`/`pointerup`, but any audio still buffered in the AudioWorklet at the exact release instant must still make it into the final segment (no truncation of the last syllable at release).

**Requirements the review must adversarially verify:**
- (a) The downsample carry state is genuinely correct across an arbitrary sequence of render-quantum boundaries — no dropped or duplicated samples, verified against a reference resample of the same input.
- (b) A tap-to-toggle segment transcribes exactly once per speech/silence cycle — no double-transcribe on a jittery VAD flip, no missed segment on a very short utterance.
- (c) The hold-to-talk release boundary does not truncate trailing audio still in the worklet's internal buffer at the exact moment of `keyup`/`pointerup` — the segment closes only after that residual is flushed.
- (d) No audio frame is ever silently dropped between the worklet and the worker (a `postMessage` failure or a full buffer degrades visibly — a toast/inline warning — never a silently shorter transcript).

### 7.2 Worker lifecycle + lazy model-load race

The STT engine loads lazily (D1/D7) and runs in a Web Worker (D4) — two independent asynchronous readiness conditions that a user's first mic-button press can easily race against.

**The concrete mechanism.** `use-voice-input.ts` must not allow `audio-capture.ts` to start recording before `stt-engine.ts`'s worker reports its model(s) fully loaded — starting capture early would buffer real audio against an engine not yet ready to consume it, forcing either a lossy drop or an unbounded buffer grow while the (first-run, ~120MB) model download completes. The hook instead gates the mic-button's enabled state on an explicit `ready` flag from the worker, showing a load-progress indicator (D7's Settings surfaces the same signal) instead of a live waveform until `ready` flips. A **concurrent-gesture guard** is needed too: a hold-to-talk press while a tap-to-toggle session is already listening (or vice versa) must be rejected or queued, never allowed to start a second overlapping capture session against the same single worker/worklet pair. Teardown (component unmount, or the user disabling voice in Settings) must cleanly close the worklet's `AudioContext`, stop the `MediaStream`'s tracks (so the browser's mic-in-use indicator actually turns off), and terminate the worker — a leaked `MediaStream` track is a real, user-visible privacy footgun (the mic stays "hot" after the feature is disabled), not just a resource leak.

**Requirements the review must adversarially verify:**
- (a) A mic-button press before the engine reports ready is a no-op (or clearly disabled), never a buffered-and-later-replayed capture.
- (b) Overlapping gestures (hold-to-talk while tap-to-toggle is active, or a second hold while the first is still releasing) cannot start two concurrent capture sessions against one worker.
- (c) Disabling voice in Settings, or unmounting the chat view, genuinely stops the `MediaStream` tracks (mic indicator off) and terminates the worker — verified by asserting `track.readyState === 'ended'` and no further worker messages are processed after teardown.
- (d) A model-load failure (network error, unsupported browser) degrades to D9's toast + text-only fallback rather than leaving the mic button stuck in a perpetual "loading" state.

## 9. Forward-items (deferred, tracked)

- **TTS voice-out** — reading assistant responses aloud; explicitly out of scope this phase (Thesis, Scope). A future phase, if pursued at all.
